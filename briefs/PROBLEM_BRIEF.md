> 给 Codex / Hermes 的架构思考输入

---

## 一、我们要解决什么问题

小爪（Linux服务器，韩国节点）上有一个已经稳定运行的**网络调研自动化 pipeline**，目前是单一管道架构。我们要把它重构为一个**可扩展的 Intelligence Center**。

**核心诉求**：从"只能跑一种调研任务"升级为"能同时服务泛用研究和项目定向研究"。

---

## 二、现有系统是什么样的

### 四个已独立运行的模块

| 模块 | 脚本 | 职责 | 当前状态 |
|------|------|------|----------|
| 搜集 | `night_research.sh` + `research_worker.py` | 搜索+抓取+提取 | ✅ 稳定运行，每晚自动执行 |
| 研究 | `research_worker.py` 内置 | 深度分析+洞察提取 | ✅ 直连MiniMax API，3-5秒/次 |
| 评分 | `score_worker_web.py` / `score_worker_pdf.py` | 质量评估+分类 | ✅ GLM-5.2评分 |
| 入库 | wiki写入 + SQLite元数据 | 知识库管理 | ✅ 28K+ wiki页面 |

### 当前架构的问题

```
direction.json → 搜集 → 研究 → 评分 → 入库
（单一输入）    （单一管道）              （单一输出）
```

1. **只能处理一种任务**：当前只有"泛用行业深挖"，无法接收项目定向brief
2. **知识库沉睡**：28K+ wiki页面没有被主动检索和利用
3. **没有项目交付能力**：调研结果只入库不输出项目报告

---

## 三、我们要做成什么样

### 目标架构："一个引擎，两种模式，三层架构"

```
┌─────────────────────────────────────────┐
│         情报交付层 (Delivery)            │
│   项目报告 / 行业简报 / 实时监控         │
├─────────────────────────────────────────┤
│         研究执行层 (Research)            │
│   Track A（泛用深挖）+ Track B（项目深挖）│
├─────────────────────────────────────────┤
│         共享引擎层 (Engine)              │
│   搜集 / 研究 / 评分 / 入库             │
└─────────────────────────────────────────┘
```

**Track A（泛用行业深挖）**：保留现有功能，定时/事件驱动，覆盖18个行业
**Track B（项目滚动深挖）**：新增，Brief驱动，按项目需求定向调研，产出项目专属知识包

### 关键约束

| 约束 | 详情 |
|------|------|
| 服务器 | Linux，8核16GB，已有pipeline运行 |
| API额度 | MiniMax M3（项目额度，主力）、SenseNova（免费Fallback）、GLM-5.2（评分） |
| 知识库 | SQLite元数据 + 文件系统wiki，28K+页面 |
| 搜索引擎 | SearXNG自建 + Bocha API + MiMo联网搜索 |
| 现有代码 | Python，不依赖框架，脚本式部署 |

### 不需要做的事

- ❌ 不需要重写现有模块（它们已经稳定运行）
- ❌ 不需要搭Web界面（CLI+飞书交付即可）
- ❌ 不需要分布式架构（单机足够）

# Intelligence Center 架构设计

## 核心理念

将小爪的网络调研pipeline从单一管道重构为 **"一个引擎，两种模式，三层架构"** 的 Intelligence Center。

## 模块定义

### 1. 搜集引擎 (Collector)

**职责**：统一的搜索+抓取+提取

**输入**：
- Track A: direction.json（行业方向列表）
- Track B: brief.json（项目brief）

**输出**：
- raw_documents/（原始抓取内容）

**技术栈**：
- SearXNG（自建搜索）
- Bocha API（中文搜索）
- MiMo联网搜索（快速）
- Playwright（网页抓取）

### 2. 研究引擎 (Researcher)

**职责**：深度分析+洞察提取

**输入**：
- raw_documents/（来自Collector）
- 知识库已有内容（用于交叉验证）

**输出**：
- findings/（研究发现）
- summaries/（本轮摘要）

**技术栈**：
- MiniMax M3（主力推理）
- SenseNova DS Flash（Fallback）

### 3. 评分引擎 (Scorer)

**职责**：内容质量评估+分类

**输入**：
- findings/（来自Researcher）
- summaries/（来自Researcher）

**输出**：
- scored_content/（评分后的内容）
- 分类结果（wiki_passed/insights/rejected）

**技术栈**：
- GLM-5.2（主力评分）
- MiniMax M3（Fallback）

### 4. 入库引擎 (Archivist)

**职责**：知识库管理+索引+链接

**输入**：
- scored_content/（来自Scorer）

**输出**：
- wiki页面（写入知识库）
- 向量索引更新
- 关键词索引更新

**技术栈**：
- SQLite（元数据）
- BAAI/bge-small-zh-v1.5（embedding）
- 正则+关键词（索引）

## 两种模式

### Track A: 泛用行业深挖

**触发方式**：
- 定时（每日/每周）
- 事件（新政策/竞品动作/市场变化）

**流程**：
1. 从direction.json读取研究方向
2. Collector搜集相关文档
3. Researcher深度分析
4. Scorer评估质量
5. Archivist入库

**特点**：
- 覆盖面广
- 深度适中
- 持续更新

### Track B: 项目滚动深挖

**触发方式**：
- Brief驱动（Allen/Emma下发brief）

**流程**：
1. 读取brief.json（项目需求）
2. 检索知识库已有内容
3. Collector搜集缺口内容
4. Researcher针对项目深度分析
5. Scorer评估+分类
6. 生成项目专属知识包

**特点**：
- 针对性强
- 深度高
- 有明确交付物

## 三层架构

### 共享引擎层

所有模块共享的基础设施：
- LLM调用链（MiniMax → SenseNova → DeepSeek）
- 搜索引擎链（SearXNG → Bocha → MiMo）
- 知识库（28K+ wiki页面）
- 元数据管理（SQLite）

### 研究执行层

Track A和Track B的具体执行逻辑：
- 任务调度（cron/手动触发）
- 并发控制（避免API限流）
- 状态管理（进度追踪/断点续跑）
- 错误处理（重试/Fallback）

### 情报交付层

面向用户的产出：
- 项目报告（飞书文档/Markdown）
- 行业简报（定期推送）
- 实时监控（异常告警）
- 知识库查询接口

## 开发计划

### Phase 1: 引擎重构（Week 1-2）

**目标**：将现有pipeline拆分为独立模块

**任务**：
1. 搜集引擎模块化（Collector）
2. 研究引擎模块化（Researcher）
3. 评分引擎模块化（Scorer）
4. 入库引擎模块化（Archivist）

**验收标准**：
- 每个模块可独立运行
- 模块间通过标准接口通信
- 现有功能100%保留

### Phase 2: 双模式支持（Week 3-4）

**目标**：支持Track A和Track B两种模式

**任务**：
1. Track A调度器（定时/事件驱动）
2. Track B调度器（Brief驱动）
3. 知识库检索接口
4. 项目知识包生成

**验收标准**：
- Track A可自动运行
- Track B可接收brief并执行
- 两种模式可并行

### Phase 3: 交付层（Week 5-6）

**目标**：面向用户的交付能力

**任务**：
1. 报告生成器（Markdown/飞书文档）
2. 监控面板（HTML看板）
3. 查询接口（API/命令行）
4. 推送机制（飞书/邮件）

**验收标准**：
- 可生成标准化报告
- 可监控pipeline状态
- 可查询知识库内容

### Phase 4: 优化迭代（Week 7+）

**目标**：持续优化

**任务**：
1. 性能优化（并发/缓存）
2. 质量优化（评分模型调优）
3. 功能扩展（新数据源/新分析维度）

## 技术决策

### 1. 为什么用模块化而不是重写？

- 现有pipeline已经稳定运行
- 模块化可以在不中断服务的情况下重构
- 降低风险，可控交付

### 2. 为什么Track A和Track B共享引擎？

- 避免重复造轮子
- 统一的质量标准
- 知识库自动积累

### 3. 为什么用飞书群做协作？

- Allen习惯飞书
- 透明可追溯
- 多agent可参与

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| API限流 | 批量任务失败 | 并发控制+队列+重试 |
| 知识库膨胀 | 存储/检索变慢 | 定期清理+索引优化 |
| Agent通信失败 | 任务卡住 | 超时机制+人工介入 |
| 代码冲突 | 开发阻塞 | Git分支+PR review |
